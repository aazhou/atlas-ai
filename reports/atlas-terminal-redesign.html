① 信息架构

先搞清楚「谁、何时、看什么、做什么」，再设计页面。

01

个人投资者 × 社群发起人。

不盯盘，不做日内高频，但需要「关键信号不过夜」。

02

三大场景

🌅 开盘前检查 — 2分钟扫完全球，决定今天盯不盯
📡 盘中监控 — 被动接收预警，信号触发才深入看

✅ 红涨绿跌，全球统一

③ 页面设计详述

每页的卡片布局、数据来源、交互行为。

Page 1：全景 Panorama（默认首页）

顶部状态条

四大市场实时快照。
格式：国旗 + 指数名 + 点位 + 涨跌幅。
绿色点=交易中，灰色点=已收盘。
数据源：A股指数API + 恒指API + SPY/QQQ + Binance BTC。
始终可见，不随页面切换消失。

预警横幅

最醒目的位置，红色左边框。
只显示最高优先级的 1-2 条信号。
点击跳转到「信号」页面查看详情。
无信号时不显示此区域，不占地方。

总资产卡片

第一大卡片。总市值 + 日盈亏 + 周盈亏。
数字大、醒目，一眼看到今天赚/亏多少。
数据源：portfolio.json 汇总计算。
红涨绿跌，颜色即信息。

跨市场持仓网格

2列 mini 卡片网格。每张卡片：代码+价格+涨跌幅。
按市场分区：A股 → 港股·美股 → 加密。
一屏看完所有持仓表现。点击跳转到持仓详情。
数据源：portfolio.json + Binance API + yfinance。

Page 2：信号 Signals

优先级分层

🔴 高优（红色）：需要立即行动 — 费率反转、OI背离等量化信号。
🟡 中优（黄色）：需要关注 — 板块资金异常流出、持仓技术破位。
🔵 提示（蓝色）：了解即可 — 均线突破、金叉等常规信号。

每条信号包含

信号标签（高/中/低优先级）
触发时间
信号标题
数据描述
💡 建议行动（仅高优信号有）

板块资金异动

Top 3 流入 + Bottom 3 流出。
数据源：fund_flows-{date}.json。
只显示极端值，中间80%不展示。
每行：板块名 + 净流入(亿) + 涨跌幅。

信号生成逻辑

量化策略自动扫描（费率反转+OI背离）。
板块资金流异常检测（>2σ偏离）。
技术指标突破/跌破（MA20、MACD金叉死叉）。
通过cron脚本生成 → 写入JSON → 前端读取。

Page 3：持仓 Positions

市场Tab切换

顶部4个Tab：A股 | 港股 | 美股 | 加密。
每个Tab下展示该市场所有持仓。
默认显示A股（龙哥主要市场）。

持仓卡片结构

名称 + 现价 + 涨跌幅（大字）
成本 + 持仓盈亏（小字）
Mini K线柱状图（最近20根）
技术指标摘要（MACD/MA/RSI）
如有预警：红色角标提示

AI研判卡片

每市场底部有一张AI分析卡。
综合该市场所有持仓的技术面、资金面、消息面。
给出持有/加仓/减仓建议。
数据源：portfolio.json中的AI分析字段。

详情展开（P2）

点击任一持仓卡片 → 展开完整K线图（Canvas绘制）。
显示日K + 均线 + 成交量。
P1 MVP中暂不做，先做静态展示。

Page 4：工具 Tools

复盘入口

链接到 review.html（每日自动生成）。
显示最新复盘摘要 + 生成时间。
收盘后自动出现，盘中不显示。

DuckDB状态

显示数据底座运行状态。
表结构概览 + 今日采集记录数。
方便龙哥确认数据采集正常。

策略回测概览

显示量化策略近期表现。
胜率、平均收益、信号频率。
点击跳转到 crypto/quant.html 详情。

系统健康面板

链接到 system.html。
Cron任务状态汇总。
部署通道状态（阿里云/Vercel/GH Pages）。

⑥ 技术实现方案

🏗️ 架构原则

  • 纯静态站 — HTML+CSS+JS，无运行时依赖
  • 数据通过JSON注入 — cron脚本定期导出JSON → 前端fetch
  • 渐进增强 — 预设数据兜底，JSON不可用时降级展示
  • 三通道部署 — 阿里云(主) + Vercel + GitHub Pages
  • DuckDB 为数据底座，JSON 仅为静态导出层

📱 移动端策略

  • 375px 首发设计，CSS Grid + Flexbox
  • 断点：480px（窄布局）、768px（2列）、1024px（3列+侧栏）
  • 底部固定 Tab 栏（移动端）/ 左侧导航（桌面端）
  • 所有卡片 max-width: 100%，不允许水平溢出
  • 字体：系统字体栈，数字用等宽

🔄 数据流

# 采集
cron 脚本(每5-10分钟)
  → DuckDB 写入
  → JSON 导出
  → git push / SCP 阿里云

# 前端
index.html (Atlas Terminal)
  → fetch('data/...json')
  → 解析 → 渲染卡片
  → 404 → 降级到预设值

与当前 atals-ai 数据架构完全一致，无需改动 cron 脚本。

⑦ vs v1 的核心差异

v1 问题

旧 index.html 只是在旧页面上删足球入口

• 5个入口卡片，无信息密度
• 没有全局状态条
• 没有预警系统
• 没有移动端优化
• 需要点3-4次才能看到关键数据

v2 全新设计

从零重新思考信息架构，不是改旧页面

• 4页Tab，每页一屏看完
• 全局状态条始终可见
• 预警信号最醒目位置
• 移动端375px首发设计
• 首屏直接看到总资产+持仓+预警

关键区别总结

维度 v1 (旧) v2 (新)
首屏信息 5个入口链接 总资产+持仓+预警+4市场快照
预警系统 无 独立信号页，优先级分层，含行动建议
移动端 桌面端缩放 375px首发，底部Tab导航
全局状态 无 顶部固定状态条，4市场实时
数据展示 链接到各子页面 首页直接展示关键数据，一键可达
信息架构 按市场分入口 按场景分层：全景→信号→持仓→工具

⑧ 关键设计决策

决策1：为什么是4个页面而不是3个或5个？
龙哥的4个核心场景（开盘检查→盘中监控→收盘复盘→深度研究）恰好对应4个页面。3个太少（无法覆盖所有场景），5个太散（持仓和工具可拆但没必要）。
决策2：为什么「全景」页直接展示持仓，而不只是市场指数？
龙哥是个人投资者，不是宏观分析师。他需要第一眼看到「我今天赚了还是亏了」，而不是「上证涨了多少」。总资产+持仓网格比指数列表更有用。
决策3：为什么信号页独立出来，而不是嵌在全景页里？
预警信号需要独立空间展示详情+历史+建议行动。嵌在全景页会挤压其他信息。独立页面 = 更好的信息密度 + 更专注的体验。高优信号通过顶部横幅在全景页可见。
决策4：为什么用底部Tab而不是顶部Tab？
移动端拇指操作区在底部。4个Tab放底部比顶部更容易单手操作。桌面端改为左侧纵向导航。
决策5：不重新发明轮子
子页面（stock/*, us/*, hk/*, crypto/*, system.html, review.html）保持现有实现不变。Terminal 是新首页+信号聚合层，不替代子页面的深度功能。导航保持双向连通。
